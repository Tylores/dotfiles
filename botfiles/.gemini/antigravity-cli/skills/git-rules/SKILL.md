# Global Git Architecture & Commit Standards

## Context
This playbook enforces strict repository hygiene across all local and remote operations. It mandates Conventional Commits and ensures clean, linear branching habits to keep the codebase history deterministic, traceable, and scannable.

## Trigger Conditions
Activate this skill automatically whenever a task involves staging files, creating branches, executing git commands, generating commit messages, or writing pre-commit / CI automation hooks.

## Core Directives

### 1. Mandatory Conventional Commits
All proposed commit messages or squash-merge descriptions generated by the agent MUST strictly conform to the Conventional Commits 1.0.0 specification. 
* **Format:** `<type>(<optional-scope>): <description>`
* **Types Allowed:**
  * `feat`: A new user-facing or architectural feature.
  * `fix`: A bug or runtime exception fix.
  * `docs`: Documentation changes only (README, inline comments).
  * `style`: Changes that do not affect the meaning of the code (white-space, formatting, linting fixes).
  * `refactor`: A code change that neither fixes a bug nor adds a feature.
  * `perf`: A code change that improves performance.
  * `test`: Adding missing tests or correcting existing tests.
  * `chore`: Updating build tasks, dependencies, package manager configs, or dotfiles.
* **Grammar Rule:** The description must be in lowercase, imperative mood, and short (under 50 characters). Do not end the description with a period.
* **Example:** `feat(http): implement value federate publication loop`

### 2. Clean Branching & Multi-Tenant Isolation
* **No Direct Mutations to Main:** The agent is strictly forbidden from committing code directly to protected primary branches (`main`, `master`, `dev`, `release`).
* **Branch Naming Standard:** When proposing or creating a new branch, names must use a slash-separated namespace matching the task type:
  * `feature/<short-descriptive-name>`
  * `bugfix/<short-descriptive-name>`
  * `chore/<short-descriptive-name>`
* **Linear History Enforcement:** Prioritize linear history. When integrating upstream changes, always use rebase methodology (`git pull --rebase` or `git rebase origin/main`) instead of generating messy merge commits.

### 3. Staging and Scope Limitations
* **Atomic Commits Only:** Keep commits small and focused on a single logical change. Do not bundle a bugfix, a test adjustment, and a formatting cleanup into a single sweeping commit block.
* **Verify Untracked Garbage:** Ensure swap files (`*.swp`), system dotfiles, compiled binaries, or caching nodes (`__pycache__/`, `target/`) are never staged or committed.

### 4. Strict Layer Isolation Check (The Inside-Out Blueprint)
Before proposing any commit message, you MUST evaluate the `git diff --cached` context against these three distinct architectural layers:
* **Layer 1 (Definition):** Purely additive structural scaffolding (e.g., new structs, types, enums, interfaces, database schemas, constants). No business logic loops.
* **Layer 2 (Implementation):** Business plumbing, data wiring, algorithms, or function logic modifying the active behavior of the app.
* **Layer 3 (Verification):** Test suites, unit tests, mock frameworks, test data fixtures (`*_test.go`, `test.rs`, etc.).

#### Layer Verification Protocol:
1.  **Detect Overlap:** Check if the staged code contains changes across more than ONE architectural layer simultaneously.
2.  **Raise Violation:** If mixing is detected (e.g., Layer 1 definition mixed with Layer 2 application loops, or Layer 2 logic mixed with Layer 3 unit tests), you MUST prepend a highly visible **`[WARNING: LAYER MIXING DETECTED]`** header to your output.
3.  **Provide Split Guidance:** Stop the commit process. List which files/lines belong to which layer, and output the exact `git restore --staged` or `gitsigns` patch guidance the user needs to run to strip out the offending lines.

## Hard Constraints (Banned Actions)
* **CRITICAL:** Do not use or suggest raw `--force` flags under any circumstances. Always use `--force-with-lease` if a remote tracking branch must be overridden.
* **CRITICAL:** Never generate generic, lazy commit messages like "fix bug," "update code," "wip," or "changes."
* **CRITICAL:** Do not stage modified configuration properties (`*.toml`, `*.json`) alongside code enhancements unless they are explicitly required for the code to compile.

## Verification Protocol
1. **Commit Dry-Run:** Before committing code on behalf of the user, output the exact planned commit command string to the chat window for visual inspection.
2. **Lint Validation:** If the repository utilizes local verification layers (like `husky`, `commitlint`, or pre-commit hooks), the agent must run or verify against those hooks before declaring a repository state as clean.
